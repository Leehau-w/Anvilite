## 一、项目概述

### 基本信息

- **项目名**：Anvilite · 锻石
- **命名由来**：Anvil（铁砧，锻造的核心工具）+ -ite（矿物后缀），寓意经过反复锤炼后炼出的精华。中文名「锻石」是发起人提出的，精准压缩了英文名的含义。
- **一句话描述**：一个RPG游戏化的桌面任务管理应用，角色是用户真实成长的镜像，帮助用户在执行长周期任务时获得即时反馈和持续动力。
- **产品形态**：Electron桌面应用（Windows优先）
- **开源策略**：完成后发布到GitHub开源

### 项目愿景

市面上不存在一款产品能同时做好「认真的任务管理」和「有深度的游戏化成长」。Habitica游戏重但管理弱；Todoist管理强但毫无成就感；Finch温暖但太轻量；Forest只解决专注不解决规划。Anvilite要填补这个空白。

### 核心公式

**认真的任务管理 × 角色即镜像 × 温和正向引导 × 完整成长记录**

## 二、项目发起人画像

了解发起人的背景和偏好对于后续沟通至关重要：

- **编程经验**：较少，但愿意学习。第一版完全通过AI编程（Claude Code）完成。
- **开发方式**：Vibe Coding——使用Claude Code命令行工具，通过自然语言指令让AI写代码。不直接手写代码。
- **审美偏好**：对细节要求高，第一版因大量细节由AI自由发挥而不满意。明确表示「每一个细节都需要讨论到位」。
- **决策风格**：喜欢在充分理解选项后做决策。需要PM提供选项分析和推荐理由，而非直接给结论。但一旦做了决策就很果断。
- **产品直觉**：相当好。多次提出超越PM建议的想法（如「千锤百炼」→「锻石」的中文名优化、世界地图三层空间架构的核心概念）。给发起人充分的创意空间很重要。
- **游戏经验**：对RPG有深入了解，希望产品有真正的养成深度，而非浅层游戏化贴皮。
- **操作系统**：Windows
- **GitHub经验**：有账号，基本Git操作不太熟，需要操作指导。

### 沟通要点

- 发起人有时会在被问到"你有想法吗"时先说没有，但实际上心里有模糊的方向。给TA空间去表达，往往会得到很好的答案。
- 发起人重视「为什么」比「是什么」。做推荐时解释理由比直接给方案更有效。
- 发起人对惩罚机制有强烈的负面态度，这是不可动摇的设计红线。
- 当发起人说"都可以"或选了所有选项，需要PM主动做工作量评估和优先级建议，避免scope creep。

## 三、问题定义与市场机会

### 核心问题

人在执行反馈周期长的任务时动力不足。典型场景：
- 学习外语（需要数月到数年的持续投入）
- 完成工作和学业中的大型项目
- 培养兴趣爱好（乐器、绘画、编程等）

这些任务的共同特点是：付出和回报之间有很长的延迟，过程中缺乏即时反馈，容易在中途放弃。

### 为什么现有产品没解决好

通过对Habitica、Todoist、Finch、Forest等产品的用户评价、应用商店评论、Reddit讨论和行业分析的综合调研，提炼出6个根本失败模式：

**根因1：游戏化不创造动力，只放大动力**
Duolingo没有激励任何人学语言——它只是帮已经想学的人坚持下去。如果用户的根本动力不存在，再多游戏元素都是空中楼阁。
→ 启示：Anvilite的核心不是「让无聊变有趣」，而是「帮用户看见自己的进步」。

**根因2：外在奖励侵蚀内在动机**
心理学研究证明过度的外在奖励会削弱内在动力。用户开始为了XP做任务而非为事情本身，一旦奖励感减弱，动力比不用app时还低。
→ 启示：奖励系统应该是「锦上添花」，核心是让用户与真实目标建立连接。

**根因3：惩罚机制与心理安全冲突**
Habitica的HP损失、角色死亡在用户状态不好时加剧焦虑，导致用户回避打开app，形成恶性循环。有用户因心理健康原因被迫退出。
→ 启示：惩罚应该是「机会成本」（展示错过了什么）而非「直接损失」（失去了什么）。

**根因4：打卡和任务管理有根本区别**
习惯打卡（每天喝8杯水）和任务管理（完成毕业论文）是完全不同的需求。市面没有产品把两者真正融合好。
→ 启示：这是市场空白——认真的任务管理 + 有深度的成长系统。

**根因5：成长天花板导致长期流失**
Habitica用户一年后感觉能做的都做完了。等级封顶、宠物收集完、装备买齐。
→ 启示：成长系统要有纵深，且与用户真实的技能成长挂钩。

**根因6：为了量化而量化，忽略了质**
用户拆碎任务刷XP、标记没做的事为已完成、优先做简单任务回避困难任务。
→ 启示：奖励机制要识别任务难度和真实价值。

### 市场机会总结

市面上不存在一款产品能同时做到：任务管理足够认真（可拆解、有截止日、有优先级）、游戏化有真正的深度（不只是积分徽章）、不惩罚用户（温和正向引导）、完整记录成长轨迹（可回看的人生存档）。这就是Anvilite的机会。

## 四、竞品深度分析

### Habitica（RPG + 习惯打卡，评分4.1/5）

**严重问题：**
- 游戏压过了工具：RPG系统太复杂，非游戏玩家觉得界面混乱
- 惩罚机制造成焦虑：扣HP、拖累队友、角色死亡制造压力和负罪感
- 任务管理薄弱：本质是习惯打卡器，不支持任务拆解和项目管理

**中等问题：**
- 内容耗尽感：一年后能做的都做完了
- 社区治理问题：公会/酒馆被移除，志愿者版主被解雇
- 技术质量差：移动端频繁崩溃、卡顿、通知失效

### Finch（宠物养成 + 自我关怀，评分4.8/5）

- 偏向情感支持而非任务管理
- 养成一只小鸟，游戏深度有限
- 完全正向激励但缺乏紧迫感

### Forest（专注计时 + 种树，评分4.6/5）

- 只解决「专注」不解决「规划」——帮你在25分钟内不碰手机，但不帮你决定做什么
- 种树成就感随时间递减

### Todoist / TickTick（专业任务管理，评分4.7/5）

- 完成任务毫无反馈感：勾选checkbox然后任务消失
- Karma系统形同虚设
- 长期项目缺乏激励

## 五、产品设计哲学

以下是经过充分讨论后确认的设计原则，视为**不可更改的红线**：

### 5.1 角色是镜像，不是游戏壳

这是Anvilite与所有竞品最根本的区别。

Habitica的箭头：**完成任务 → 游戏角色变强**（角色成长与用户现实脱节）
Anvilite的箭头：**你变强 → 角色映射出你的成长**（角色是用户真实成长的投影）

具体体现：学了三个月英语，角色的语言区域从荒芜变繁荣；坚持跑步一年，竞技场从简陋变宏伟。这不是游戏数值，是用户真实成长的可视化。

### 5.2 任务管理为本，游戏化为翼

先做好任务创建、拆解、优先级、截止日——这是地基。游戏元素（XP、等级、区域繁荣）作为完成任务的副产品自然出现。理论上用户可以关闭游戏化层，底下是一个好用的任务管理器。

### 5.3 温和引导，绝对拒绝惩罚

这是发起人最坚定的立场，没有任何商量余地：
- 不存在HP损失、角色死亡、连击中断惩罚
- 中断用5种温和状态表达（充电中/休憩/远行等），每种都用正向文案
- 低谷期自动降低期望而非加码惩罚
- 展示机会成本（「完成这3个任务就能到Lv5」）而非直接损失

### 5.4 进步可视化 > 奖惩驱动

核心反馈不是「+50XP」这样的抽象数字，而是让用户看到自己真实的投入和进步——投入了多少时间、完成了多少任务、哪些领域在成长。XP和等级只是辅助的量化手段。

### 5.5 完整记录成长轨迹

所有事件全量记录——完成、中断、恢复、里程碑，形成用户可回看的「人生存档」。中断和重新开始也被记录，不是作为失败，而是作为成长故事的一部分。半年后回看时间线，那种「原来我已经走了这么远」的感觉是产品的核心情感价值。

### 5.6 抗作弊的奖励设计

XP与任务难度强挂钩（一个高难度任务 > 十个简单任务），防止用户拆碎任务刷XP。周期性回顾帮用户反思是否在做重要的事。

## 六、产品架构设计

### 6.1 导航结构

四个主Tab：**仪表盘 / 任务 / 世界地图 / 时间线**

原来的「角色」Tab被「世界地图」取代。角色属性（等级、雷达图、称号）移入里程碑殿堂，通过一个图标入口查看。

### 6.2 世界地图三层空间架构

这是第二版最重大的架构创新，由发起人提出。核心理念：**用户打开Anvilite看到的不是一个工具界面，而是一个活着的、属于自己的小世界。**

**第一层：世界地图（主视角）**
- 打开「世界地图」Tab默认看到的
- 等距像素风的俯瞰视角小世界
- 6个默认区域（见下表），每个对应一个任务分类
- 区域繁荣度随任务完成综合变化：建筑升级 + 环境变化 + 装饰增加
- 用户可增删区域（里程碑殿堂除外）、可修改区域名称

**第二层：区域内部空间**
- 点击世界地图上的区域进入
- 场景中展示角色在该空间里的状态（如学习时在书房里）
- 包含该分类下的任务列表
- 包含该领域的成长数据面板（投入时长、完成数、技能等级）
- 包含该领域的装饰/收藏展示（完成任务解锁的物品）

**第三层：里程碑殿堂**
- 固定区域，不可删除不可改名
- 展示所有里程碑和徽章成就
- 非简单2D方形贴图，要有空间感但不需要3D
- 包含角色属性查看入口（等级、能力雷达图、称号体系）
- 里程碑包括用户自定义里程碑和系统事件徽章

### 6.3 默认区域定义

| 区域名称 | 对应分类 | 内部空间意象 | 可删除 | 可改名 |
|---------|---------|------------|--------|--------|
| 智慧塔 | 学习 | 书房/图书馆 | ✓ | ✓ |
| 工坊 | 工作 | 工匠作坊 | ✓ | ✓ |
| 竞技场 | 运动 | 训练场 | ✓ | ✓ |
| 星光阁 | 兴趣爱好 | 创意工作室 | ✓ | ✓ |
| 家园 | 生活/日常 | 温馨小屋 | ✓ | ✓ |
| 里程碑殿堂 | 成就展示 | 荣誉殿堂 | ✗ | ✗ |

### 6.4 角色状态系统

5种温和状态，通过多样化避免单一状态变成惩罚感：

| 状态 | 触发条件 | 视觉 | 文案 |
|------|---------|------|------|
| active 活跃 | 近期完成任务 | 精神焕发，活跃光效 | （无，正常状态） |
| charging 充电中 | 1-3天未活跃 | 安静阅读/冥想 | 好好充电，随时准备出发 |
| resting 休憩 | 3-7天未活跃 | 星空下安睡 | 好好休息，蓄力再出发 |
| traveling 远行 | 7天以上未活跃 | 背影望向远方 | 无论多久，这里都等你回来 |
| returning 回归 | 中断后重新完成任务 | 带着旅行痕迹回来 | 欢迎回来，冒险者 |

状态转换规则：
- 任意状态 → active：完成任何一个任务
- active → charging：距离上次活跃超过1天
- charging → resting：距离上次活跃超过3天
- resting → traveling：距离上次活跃超过7天
- 非active → returning → active：完成任务时先短暂显示returning再切换到active

### 6.5 XP与等级系统（框架已定，细节待讨论）

**已确认的框架：**
- 基础XP = 难度 × 系数
- 加成类型：按时完成、连续活跃、高难度额外奖励
- 等级不封顶
- 升级所需XP递增
- 连击中断不做任何惩罚，只是天数归零

**待讨论的细节：**
- 具体的系数数值
- 等级名称体系（如Lv1-5叫什么、Lv6-10叫什么）
- 升级曲线的具体公式

### 6.6 成长记录策略

- **全量记录**：所有事件全部记录到时间线
- **成长叙事**：当前版本不做，后续用AI生成个性化叙事总结
- **日志筛选**：当前版本全部显示，后续用AI智能筛选关键节点

## 七、第一版原型总结与教训

### 7.1 第一版做了什么

使用Claude Code进行AI编程，约2小时完成了可运行的原型：
- 四页面布局（仪表盘/任务/角色/时间线）+ 侧边栏导航 + 底部状态栏
- 任务CRUD + 分类筛选（5个分类）
- XP计算 + 等级系统（测试中达到Lv.2 初心者）
- 角色状态（充电中状态已实现）
- 能力雷达图（学习/兴趣两个维度）
- 成长时间线（4条记录：任务完成、升级、回归）
- 连续活跃统计

### 7.2 第一版的问题

1. **PRD细节不足导致AI自由发挥**：字体、间距、动画、颜色等大量细节由AI自行决定，质感不够
2. **养成感薄弱**：角色只是一张占位图标+数据展示，和Habitica本质区别不大
3. **缺乏空间感**：角色页面是平面信息展示，没有"世界"的感觉
4. **视觉偏"程序员风"**：信息都在但缺乏设计感和层次感
5. **品牌名未更新**：界面仍显示旧名称

### 7.3 核心教训

> **PRD写到80分，AI会用自己的"审美"填补剩下的20分，而那20分往往是决定产品质感的关键。**

第二版的核心策略：把每一个细节讨论到位，产出一份AI无法"自由发挥"的开发指令文档。讨论完成标准是「开发者看完文档后零歧义，不需要做任何设计决策」。

## 八、第二版重构方向

### 8.1 重构范围

从零开始完全重做，不复用第一版代码。保留第一版验证过的核心逻辑（任务CRUD、XP计算、等级系统），但所有UI、交互、视觉全部重新设计。

### 8.2 工作方法

采用「一个模块走到底」的讨论方式：选定一个模块，从数据结构→交互流程→视觉设计→动画细节→开发指令，全部敲定到像素级细节后，再进入下一个模块。

每个模块讨论完成后产出的是一份精确到「AI无法自由发挥」的开发指令文档，发起人拿去直接喂给Claude Code执行。

### 8.3 视觉风格

- **整体风格**：等距（Isometric）像素风
- **角色**：2D插图，使用AI生成素材
- **主题**：暗色主题
- **素材方案**：待设计细化到具体视觉时再确定

候选素材方案：
- A. AI生成像素素材（速度快但一致性难保证）
- B. 开源像素素材包为基础（一致性好但受限于已有风格）
- C. 程序化生成+少量手绘（繁荣度变化开发友好但像素感弱）
- D. 混合方案（程序化布局+素材包建筑+AI场景，推荐但尚未确认）

### 8.4 素材工作量估算

世界地图6区域 × 约4级繁荣度 = 约20套区域外观 + 6个内部空间场景 + 角色5种状态sprite + 装饰物件/徽章/UI图标 = 粗估80-120张像素素材。这是整个项目工作量最大的非编码部分。

## 九、技术背景

### 9.1 技术栈

| 技术 | 用途 | 选型理由 |
|------|------|---------|
| Electron | 桌面应用框架 | 跨平台，未来可转Web |
| React 18 + TypeScript | UI开发 | 生态丰富，类型安全 |
| Zustand | 状态管理 | 轻量简洁，学习成本低 |
| Framer Motion | 动画系统 | 完成反馈和过渡动画 |
| electron-store | 本地持久化 | JSON存储，后续可迁移SQLite |
| Tailwind CSS | 样式系统 | 快速开发，一致性好 |

### 9.2 开发环境

- 操作系统：Windows
- AI编程工具：Claude Code（命令行）
- 版本管理：Git + GitHub
- 仓库名：`anvilite`

### 9.3 开发流程

发起人采用Vibe Coding方式：通过自然语言向Claude Code下达指令，Claude Code直接创建/编辑代码文件。PM的核心产出物是**足够精确的开发指令文档**，而非传统PRD。

### 10.2 每个模块的讨论标准

讨论完成的标志是产出一份满足以下条件的文档：

1. **零歧义**：任何两个开发者（或AI）读完文档后会做出一样的东西
2. **不留决策空间**：不能有「可以考虑」「建议」等模糊措辞，必须是确定的指令
3. **包含边界情况**：空状态怎么显示、极端数据怎么处理、错误状态怎么提示
4. **视觉可量化**：颜色有色值、间距有像素值、动画有时长和缓动曲线
5. **有交互流程图**：用户从A到B的每一步都有定义
