## 四、F1 — 任务计时

### 4.1 产品目标

让用户直观看到当前任务持续了多久，完成时自动记录实际用时到 `actualMinutes`，免去手动填写的麻烦。

### 4.2 数据模型变更

**Task 新增字段：**

```typescript
interface Task {
  // ... 已有字段 ...
  timerStartedAt: string | null   // 计时开始的 ISO 时间戳，null 表示未在计时
  timerAccumulated: number        // 暂停前已累计的秒数，默认 0
}
```

**Habit 新增字段：**

```typescript
interface Habit {
  // ... 已有字段（已有 timerStartedAt）...
  timerAccumulated: number        // 新增：暂停前已累计的秒数，默认 0
}
```

**计算逻辑（纯函数，放 `src/engines/timerEngine.ts`）：**

```typescript
/** 返回当前已流逝的总秒数 */
function getElapsedSeconds(startedAt: string | null, accumulated: number): number {
  if (!startedAt) return accumulated
  const elapsed = Math.floor((Date.now() - new Date(startedAt).getTime()) / 1000)
  return accumulated + Math.max(0, elapsed)
}

/** 格式化为 HH:MM:SS 或 MM:SS */
function formatElapsed(totalSeconds: number): string {
  const h = Math.floor(totalSeconds / 3600)
  const m = Math.floor((totalSeconds % 3600) / 60)
  const s = totalSeconds % 60
  if (h > 0) return `${h}:${String(m).padStart(2, '0')}:${String(s).padStart(2, '0')}`
  return `${m}:${String(s).padStart(2, '0')}`
}
```

### 4.3 Store 变更

**taskStore 新增 action：**

```typescript
startTimer: (taskId: string) => void
// 设置 timerStartedAt = now, 不清空 timerAccumulated

pauseTimer: (taskId: string) => void
// timerAccumulated += (now - timerStartedAt), timerStartedAt = null

stopTimer: (taskId: string) => void
// 同 pauseTimer，但额外更新 actualMinutes = Math.round(timerAccumulated / 60)
// 然后重置 timerAccumulated = 0, timerStartedAt = null
```

**completeTask 增强：**
完成任务时，如果 `timerStartedAt !== null`（正在计时），自动执行 `stopTimer` 逻辑，将累计时间写入 `actualMinutes`。如果 `timerAccumulated > 0`（已暂停但有累计），同样写入。

**habitStore 同步变更：**
已有 `startHabitTimer` / `pauseHabitTimer`，新增 `timerAccumulated` 字段支持，逻辑同上。

> **TD 意见 F1-3（修复现有 bug）**：当前 `pauseHabitTimer` 直接清空 `timerStartedAt` 导致计时数据丢失。开发时必须同步改造为"先累加到 `timerAccumulated` 再清空"。

### 4.4 应用关闭/休眠的边界策略

> **TD 意见 F1-2 + PM 确认**

**问题场景**：用户开着计时器关闭 Electron / 电脑休眠过夜 / 隔天打开应用，`timerStartedAt` 持久化在 localStorage，会显示荒谬的超长时间。

**策略**：在 `taskStore.onRehydrateStorage` 和 `habitStore.onRehydrateStorage` 中：

```typescript
// 4 小时阈值（秒）
const TIMER_STALE_THRESHOLD = 4 * 60 * 60

function autoStaleTimers(tasks: Task[]): { tasks: Task[], staleCount: number } {
  let staleCount = 0
  const now = Date.now()
  const updated = tasks.map(task => {
    if (!task.timerStartedAt) return task
    const elapsed = (now - new Date(task.timerStartedAt).getTime()) / 1000
    if (elapsed > TIMER_STALE_THRESHOLD) {
      staleCount++
      return {
        ...task,
        timerAccumulated: task.timerAccumulated + TIMER_STALE_THRESHOLD,
        timerStartedAt: null,
      }
    }
    return task
  })
  return { tasks: updated, staleCount }
}
```

- 超过 4 小时的计时器自动暂停，累计时间记为 4 小时（阈值本身，而非荒谬的超长值）
- 自动暂停后，下次打开应用弹 toast 提示：「N 个计时器已自动暂停（离开超过 4 小时）」
- habitStore 同理

### 4.6 交互规格

#### 任务卡片上的计时交互

**空闲态（未计时）：**
- 任务卡片 hover 时，在操作栏（右侧）显示计时图标 `⏱`
- 点击开始计时

**计时中：**
- 卡片上**常显**一个计时徽章，位置在卡片右上角（不需要 hover）
- 徽章内容：`⏱ MM:SS`，实时跳动（每秒更新）
- 徽章背景：`var(--color-accent)` 10% 透明度，文字 `var(--color-accent)`
- 徽章左侧有一个微弱的脉冲动画圆点（表示正在计时）
- 点击徽章 → 暂停计时
- 角色状态自动判定不受影响（已有 doing 状态逻辑）

**暂停态：**
- 徽章仍然常显，但颜色变为 `var(--color-text-dim)`
- 脉冲动画停止
- 显示已累计时间（静止不动）
- 点击徽章 → 继续计时

**完成任务时：**
- 如果有活跃计时 → 自动停止并写入 `actualMinutes`
- 如果有暂停的累计时间 → 同样写入 `actualMinutes`
- XP 飘字反馈正常触发，不受计时影响

#### 多任务并行计时

- 允许多个任务同时计时，互不干扰
- 每个任务独立维护自己的 `timerStartedAt` 和 `timerAccumulated`

#### 习惯计时

- 习惯卡片的已有计时按钮保持不变
- 新增 `timerAccumulated` 支持暂停/继续
- 完成习惯时的 `actualMinutes` 逻辑同任务

## 五、F2 — 标签系统

### 5.1 产品目标

提供轻量级的自由分类能力，补充现有区域分类（category）无法覆盖的场景：标记长期事件（如"搬家"）、心情（如"低落"）、临时分组（如"本周重点"）。

### 5.2 设计原则

- 标签是**补充**，不替代区域分类
- 一个任务可以有**多个标签**
- 标签**自由创建**，无预定义列表
- **不在仪表盘显示**，只在任务系统内可见
- 颜色自动分配（基于标签名 hash），用户无需管理颜色

### 5.3 数据模型变更

**Task 新增字段：**

```typescript
interface Task {
  // ... 已有字段 ...
  tags: string[]    // 标签数组，默认空数组
}
```

**标签颜色（纯函数，放 `src/utils/tagColor.ts`）：**

```typescript
/** 10 色调色板，覆盖主要色相 */
const TAG_PALETTE = [
  '#e8600a', '#8b5cf6', '#16a34a', '#0891b2', '#d97706',
  '#ec4899', '#6366f1', '#14b8a6', '#f43f5e', '#84cc16',
]

/** 基于标签名的稳定 hash 取色 */
function getTagColor(tagName: string): string {
  let hash = 0
  for (let i = 0; i < tagName.length; i++) {
    hash = ((hash << 5) - hash + tagName.charCodeAt(i)) | 0
  }
  return TAG_PALETTE[Math.abs(hash) % TAG_PALETTE.length]
}
```

> **TD 备注**：标签建议列表（遍历所有任务去重）在当前数据规模下无性能问题。v0.4 后端化后如任务量增大，可在 taskStore 中维护 `allTags: string[]` 缓存字段。现在不需要做，记录在案。

## 六、F3 — SOP 模块增强

v0.3.1 的 SOP 改动为 6 个子项。

### 6.1 F3-a：类型系统改造 — 从固定类型到显示风格

#### 变更必要性（PM 论证，TD 已批准）

现有 4 类型系统存在根本设计问题，不是优化而是纠正：

1. **checklist 和 itemlist 代码几乎一模一样**（差异仅编号 vs 圆点），说明产品层面无本质区别
2. **用户创建 SOP 时被迫选择类型**，但"检查型"和"清单型"语义接近，造成困惑
3. **类型一旦选定不能改**，实际使用中用户可能发现某个 SOP 更适合另一种显示方式
4. **执行模式被人为限制在 checklist/itemlist**，但所有 SOP 都有"对照着做"的需求

趁用户数据还少、迁移成本最低的时候修正。

#### 设计方案

**取消**现有的 4 种固定类型（schedule / workflow / checklist / itemlist），改为 3 种**可切换的显示风格**：

| 显示风格 | 标识 | 视觉 | 适用场景 |
|---------|------|------|---------|
| `numbered` | 编号 | 1. 2. 3. | 有序流程、操作步骤 |
| `bullet` | 圆点 | • • • | 无序清单、物品列表 |
| `timeline` | 时间轴 | 07:00 08:30 | 日程、时间安排 |

**核心变化：**
- 创建 SOP 时不再选"类型"，默认为 `numbered`
- 随时可通过内容区顶部的风格切换按钮组（3 个图标按钮）切换
- **执行模式对所有 SOP 生效**，不再限于 checklist/itemlist

#### 数据模型变更

```typescript
interface SOP {
  id: string
  folderId: string
  title: string
  displayStyle: 'numbered' | 'bullet' | 'timeline'  // 替换原 type 字段
  steps: SOPStep[]
  isSystem: boolean
  lastUsedAt: string | null
  sortOrder: number
  createdAt: string
  updatedAt: string
}
```

#### 迁移规则

| 旧 type | 新 displayStyle |
|---------|----------------|
| `workflow` | `numbered` |
| `checklist` | `numbered` |
| `itemlist` | `bullet` |
| `schedule` | `timeline` |

#### UI 变更

- 删除 `SOPChecklistView.tsx`、`SOPItemListView.tsx`、`SOPWorkflowView.tsx`、`SOPScheduleView.tsx`
- 统一为一个 `SOPStepListView.tsx`，根据 `displayStyle` 渲染不同前缀（编号/圆点/时间）
- SOPEditor 中类型选择改为风格切换按钮组

#### 执行模式统一

所有 SOP 的底部操作栏都显示「开始执行」按钮。执行模式下：
- 每个步骤前出现勾选框
- 顶部显示进度条 `3/12 已完成`
- 已勾选步骤文字变灰+删除线
- 退出执行模式时勾选状态重置

#### 风格切换按钮组

位置：SOP 内容区标题右侧

```
[标题]                    [≡] [•] [⏰]
```

三个图标按钮，当前选中的高亮（`var(--color-accent)` 底色）。点击即切换，实时保存。

### 6.2 F3-b：富文本编辑器（Tiptap，不含图片）

> **TD 原始意见**：推迟 Tiptap 到 v0.4。**PM 折中决策**：引入 Tiptap 但不做图片插入。代码块、表格、链接等都是纯文本结构，不会有 localStorage 容量问题。图片功能等 v0.4 后端 CDN 就位后再加。

#### 技术方案

引入 Tiptap 编辑器（基于 ProseMirror），所需包：

```
@tiptap/react
@tiptap/starter-kit              # 基础：段落/粗体/斜体/删除线/代码
@tiptap/extension-table           # 表格
@tiptap/extension-code-block-lowlight  # 代码块（语法高亮）
@tiptap/extension-link            # 链接
@tiptap/extension-placeholder     # 占位符文字
lowlight                          # 代码高亮引擎
```

**不引入**：`@tiptap/extension-image`（推迟到 v0.4）

#### 数据模型变更

```typescript
interface SOPStep {
  id: string
  title: string              // 保留：步骤标题（纯文本，用于执行模式/转任务）
  content: JSONContent | null // 新增：Tiptap JSON 格式的富文本内容（替换 note + warning）
  time: string | null         // 保留：仅 timeline 风格使用
  sortOrder: number
  childSteps: SOPStep[]
}

// 删除的字段：note, warning（迁移到 content 中）
```

`JSONContent` 是 Tiptap 的标准 JSON 输出格式，结构化存储富文本内容。纯文本内容体积与原 string 字段相当，不会有 localStorage 容量问题。

#### 迁移规则

将旧的 `note` 和 `warning` 字段迁移到 `content`：

```typescript
function migrateStepContent(step: OldSOPStep): SOPStep {
  const blocks: JSONContent[] = []

  if (step.warning) {
    blocks.push({
      type: 'callout',
      attrs: { variant: 'warning' },
      content: [{ type: 'paragraph', content: [{ type: 'text', text: step.warning }] }]
    })
  }

  if (step.note) {
    blocks.push({
      type: 'callout',
      attrs: { variant: 'info' },
      content: [{ type: 'paragraph', content: [{ type: 'text', text: step.note }] }]
    })
  }

  return {
    id: step.id,
    title: step.title,
    content: blocks.length > 0 ? { type: 'doc', content: blocks } : null,
    time: step.time,
    sortOrder: step.sortOrder,
    childSteps: step.childSteps.map(migrateStepContent),  // 递归迁移子步骤
  }
}
```

### 6.7 F3 迁移方案

在 `sopStore` 的 `onRehydrateStorage` 中执行，版本标记 `SOP_MIGRATION_V031`：

```typescript
function migrateSOPsToV031(state: SOPState): SOPState {
  // 1. type → displayStyle
  const sops = state.sops.map(sop => ({
    ...sop,
    displayStyle: sop.type === 'schedule' ? 'timeline'
                : sop.type === 'itemlist' ? 'bullet'
                : 'numbered',
    // 删除 type 字段
  }))

  // 2. step.note/warning → step.content (Tiptap JSONContent)
  const migratedSops = sops.map(sop => ({
    ...sop,
    steps: sop.steps.map(migrateStepContent),  // 递归迁移，含子步骤
  }))

  // 3. folder.parentId 默认 null
  const folders = state.folders.map(f => ({
    ...f,
    parentId: f.parentId ?? null,
  }))

  return { ...state, sops: migratedSops, folders }
}
```

## 八、修复项

### 8.1 X1 — 应用图标未生效

**现象**：使用 NSIS 安装包安装后，任务栏和桌面快捷方式仍显示 Electron 默认图标。

**修复方案**：
1. 使用工具从 `build/icon.png` 生成标准 `build/icon.ico`（包含 16/32/48/256 四个分辨率）
2. `electron-builder.json` 的 `win.icon` 改为 `build/icon.ico`
3. `electron/main.ts` 中 `BrowserWindow` 的 `icon` 路径同步改为 `.ico`
4. 在 NSIS 配置中添加 `"installerIcon": "build/icon.ico"`

> **TD 备注**：需确认 `build/icon.png` 源文件分辨率足够（至少 256×256）。

### 8.2 X2 — 编辑抽屉点击添加子项时自动关闭

**现象**：在编辑抽屉中点击「+」添加子项按钮，抽屉意外关闭。

**根因**：`Drawer.tsx:23` 的背景遮罩层绑定 `onClick={onClose}`，按钮点击事件冒泡触发关闭。

**修复方案**（TD 推荐方案一）：
遮罩层 `onClick` 改为仅在 `target === currentTarget` 时触发关闭：

```typescript
onClick={(e) => { if (e.target === e.currentTarget) onClose() }}
```

### 8.3 X3 — 切换分类 Tab 时任务卡片飞出动画

**现象**：切换分类 Tab 时被筛选掉的任务卡片播放向右飞出动画。

**根因**：`TaskList.tsx` 的 doing/todo 区域渲染 TaskItem 时未传 `compact` 属性，exit 动画走了 fly-out 分支。

**修复方案**（TD 推荐方案 A）：
在 `Reorder.Group` 或其父容器加上 `key={activeCategory}`，切换分类时整个列表重渲染，不触发单项退出动画。

### 8.4 X4 — 子项选框完成后样式不统一

**现象**：子项使用原生 `<input type="checkbox">`，完成后样式依赖浏览器默认，与父任务自定义 CheckButton 风格不一致。

**修复方案**（TD 推荐方向）：
提取共享 `Checkbox` 组件：

- 未完成：空心圆/方框，边框 `var(--color-border)`
- 完成：实心填充 `var(--color-success)`，白色勾号
- 尺寸：13×13px（子项）vs 18×18px（父任务），保持比例差异但风格统一
