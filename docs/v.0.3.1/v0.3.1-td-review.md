## 一、批准项（可直接进入开发）

### F1 任务计时 — 批准，附 3 条修改意见

**意见 1：必须处理 taskStore 的旧迁移冲突**

当前 `src/stores/taskStore.ts:415-416` 有这段代码：

```typescript
if ('timerStartedAt' in legacy) delete legacy.timerStartedAt
if ('timerElapsed' in legacy) delete legacy.timerElapsed
```

v0.3 时明确清理了旧版的 timer 字段。现在 F1 要重新加回 `timerStartedAt`，**必须先移除这段删除逻辑**，否则新加的字段会在 rehydrate 时被自己的迁移代码删掉。开发文档中需要标注这个前置步骤。

**意见 2：缺少应用关闭/休眠的边界策略**

PRD 未指定以下场景：

- 用户开着计时器关闭 Electron
- 电脑休眠 8 小时后唤醒
- 隔天打开应用

`timerStartedAt` 持久化在 localStorage，以上场景会显示 `8:00:00` 甚至更长的荒谬数值。

**建议方案**：在 `taskStore.onRehydrateStorage` 中，如果 `timerStartedAt` 距今超过 **4 小时**，自动暂停（将已流逝时间累加到 `timerAccumulated`，清空 `timerStartedAt`）。4 小时阈值足够覆盖正常长时间任务，同时排除过夜/忘关的场景。habitStore 同理。

需要 PM 确认此策略。

**意见 3：`pauseHabitTimer` 现有实现有 bug**

当前代码（`src/stores/habitStore.ts:154-155`）：

```typescript
pauseHabitTimer: (id) => {
  get().updateHabit(id, { timerStartedAt: null })
},
```

暂停时直接清空 `timerStartedAt`，**计时数据全部丢失**。F1 的 `timerAccumulated` 字段正好修复了这个问题。开发时需要同步改造 `pauseHabitTimer`，使其在清空 `timerStartedAt` 前先累加已流逝时间到 `timerAccumulated`。

## 二、需要拆分的项：F3 SOP 增强

### 核心问题：SOP 模块刚建好就全面重构

SOP 模块在 v0.3.0 中从零搭建，包含 4 种类型、8 个系统预设、9 个组件文件、6 个测试文件。**这些代码全部是 2 天前刚写的**。现在 F3 要：

1. 废弃刚建好的 4 类型系统，改成 3 种显示风格
2. 删除刚建好的 4 个视图组件（Checklist/ItemList/Workflow/Schedule View）
3. 引入 8 个新 npm 包（Tiptap 全家桶）
4. 重写步骤数据模型（`note/warning` → Tiptap JSONContent）
5. 添加递归子文件夹 + 右键菜单 + 跨容器拖拽

这不是"增强"，是**重写**。按前任 TD 的经验，PM 倾向于把太多东西塞进一个版本，技术总监需要控制范围。

### 逐项评估

| 子项 | 判断 | 理由 |
|------|------|------|
| **F3-a 类型→显示风格** | **可以做，但要确认必要性** | 产品判断。从技术角度，改动面大但都是替换，不引入新复杂度。如果 PM 确认 4 类型设计有根本问题而非"想优化"，可以做。 |
| **F3-b Tiptap 富文本** | **强烈建议推迟到 v0.4** | 见下方详述 |
| **F3-c 子步骤编辑 UI** | **可以做** | 当前 `childSteps: []` 写死空数组确实是功能缺失。递归编辑 UI 复杂度可控。 |
| **F3-d 链接流程** | **推迟到 F3-b 之后** | 依赖 Tiptap 自定义 inline node，没有 Tiptap 就没有载体 |
| **F3-e 子文件夹+右键菜单+拖拽** | **拆分：子文件夹+右键菜单可做，拖拽推迟** | 见下方详述 |
| **F3-f 创建时新建文件夹** | **可以做** | 小改动，依赖 F3-e 的 `parentId` 字段 |

### F3-b 推迟的理由

1. **localStorage 容量炸弹**：TD 交接文档 §4.1 明确标记 growthEvent 只增不减已是容量隐患。现在要往 SOP 数据里塞 base64 图片（单张上限 500KB）。一个包含 3-4 张截图的 SOP 可以消耗 1-2MB。localStorage 总共 5MB。**在 v0.4 引入后端存储之前，不应该增加大体积数据写入。**

2. **不可逆迁移**：`note/warning` → Tiptap JSONContent 是单向迁移。如果 Tiptap 集成出问题（性能、兼容性、包体积），回退需要再写一次反向迁移，且可能丢失富文本格式信息。

3. **投入产出比低**：SOP 模块是 v0.3 的新模块，还没有真实用户数据和使用反馈。在没有验证"用户需要在 SOP 步骤中放表格/图片/代码块"之前，就引入一个 375KB 的编辑器框架，是提前优化。

4. **对开发周期的影响**：PM 估 F3-b 为 5-7 天，是 v0.3.1 总工期的 1/3。Tiptap 自定义扩展（callout block、sopLink node）的调试周期在 PM 估算中往往被低估。

**建议**：v0.3.1 的 SOP 步骤保留 `note/warning` 字符串字段不动。等 v0.4 后端存储就位后，再引入 Tiptap + 图片上传到 CDN，一步到位。

### F3-e 拖拽推迟的理由

跨容器拖拽（SOP 从文件夹 A 拖到文件夹 B、文件夹悬停 500ms 自动展开）需要引入专门的拖拽库（dnd-kit 或类似方案），Framer Motion 的 Reorder 只支持同级重排。这是额外的新依赖和复杂度。

**建议**：v0.3.1 先用右键菜单的「移动到...」实现跨文件夹移动，拖拽留到后续版本。
