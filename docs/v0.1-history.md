# Anvilite v0.1 — 历史开发文档

> **用途**：v0.1 完整开发历程的精炼存档，供后续版本参考背景。
> **原始资料**：`docs/` 目录下的各模块规格文档、进展报告、PM 交接文档及技术总结均为本文来源，原文件保留不删除。

## 一、项目起源

### 1.1 为什么做这个产品

核心问题：人在执行反馈周期长的任务时（学语言、完成大项目、培养爱好）动力不足。通过对 Habitica、Todoist、Finch、Forest 的竞品分析，归纳出六个市场失败模式：

1. 游戏化只能放大已有动力，无法凭空创造
2. 过度外在奖励侵蚀内在动机
3. 惩罚机制在用户低谷期加剧焦虑（Habitica 扣 HP/角色死亡）
4. 打卡和任务管理是根本不同的需求，无产品真正融合
5. 等级封顶、内容耗尽导致最忠实用户最先流失
6. 拆碎任务刷 XP 的作弊问题

**市场空白**：同时做好「认真的任务管理」+「有深度的游戏化成长」+「不惩罚用户」+「完整记录成长轨迹」。

### 1.2 核心公式

**认真的任务管理 × 角色即镜像 × 温和正向引导 × 完整成长记录**

角色和 Habitica 最根本的区别：Habitica 是「完成任务 → 游戏角色变强」；Anvilite 是「你变强 → 角色映射出你的成长」。

### 1.3 开发方式

Vibe Coding：通过 Claude Code 命令行工具以自然语言指令编写代码。第一版约 2 小时完成原型，但因 PRD 细节不足导致大量细节由 AI 自由发挥，质感不够。**核心教训**：PRD 写到 80 分，AI 会用自己的审美填补剩下的 20 分，而那 20 分往往是决定产品质感的关键。

第二版采用「一个模块走到底」的方式：每个模块从数据结构到像素级细节全部讨论锁定，再开始编码。

## 二、不可违反的设计红线

这些原则在整个 v0.1 开发过程中经过反复确认，是产品的设计底线：

1. **不做任何惩罚机制** — 不存在 HP 损失、角色死亡、连击惩罚
2. **角色必须是镜像** — 不是脱离现实的游戏数值，而是用户真实成长的投影
3. **矿石不收回** — 撤销/删除任务时 XP 收回但矿石保留（努力的物质证明）
4. **紧急优先级固定红色** — `#dc2626` 跨所有主题不变
5. **全量记录 GrowthEvent** — 不自动清理，后续用 AI 筛选关键节点
6. **温和中断表达** — 5 种正向状态（充电/休憩/远行/回归/活跃），不制造焦虑

## 三、产品架构

### 3.1 导航结构

4 个主 Tab（侧边栏 56px）+ TopBar（44px）+ StatusBar（32px）：

```
🏠 仪表盘（默认）
📝 任务
🗺️ 世界地图
🏛️ 里程碑殿堂
```

> **注**：原始设计是「时间线」作为第 4 个 Tab，后在 PRD Patch 2 中改为「里程碑殿堂」，时间线（时光卷轴）移入档案馆内部。

### 3.2 世界地图三层空间

这是第二版最重大的架构创新，由发起人提出，目的是给成长系统增加空间感和养成深度：

**第一层 — 世界地图**：等距像素风俯瞰视角，多个区域对应不同任务分类，区域繁荣度随任务完成综合变化（建筑升级 + 环境变化 + 装饰增加）。

**第二层 — 区域内部空间**：点击区域进入，场景中展示角色状态、该分类任务列表、成长数据面板、装饰/收藏。

**第三层 — 档案馆**：世界地图上的固定特殊区域（原设计为里程碑殿堂），最终改为数据统计可视化中心，包含时光卷轴（时间线）。繁荣度由角色总等级驱动。

**里程碑殿堂**（侧边栏 Tab）：成就展示、能力雷达图、称号体系、徽章墙。

### 3.3 区域配置

| 类型 | 数量 | 说明 |
|------|------|------|
| 默认预设区域 | 6 个 | 家园/竞技场/书阁高塔/灵感工坊/锻造坊 + 档案馆（固定） |
| 隐藏模板 | 6 个 | 泉水/议事厅/远征大厅/观测站/植物园/广场 |
| 区域上限 | 12 个 | 含固定的档案馆 |
| 用户可建空白区域 | ✓ | 无繁荣度视觉 |

繁荣度 6 级：荒芜 → 聚落 → 丰饶 → 繁荣 → 鼎盛 → 辉煌（辉煌级全部融入锻石元素）。

## 四、核心系统规格（已锁定）

### 4.1 XP 与矿石系统

**基础 XP**（矿石与 XP 1:1 同时获得）：

| 难度 | XP/矿石 | 场景示例 |
|------|---------|---------|
| ★ | 1 | 整理桌面、回复邮件 |
| ★★ | 2 | 背 20 个单词、拉伸 10 分钟 |
| ★★★ | 3 | 写读书笔记、跑步 30 分钟 |
| ★★★★ | 5 | 写一篇文章、完成功能模块 |
| ★★★★★ | 8 | 通过考试、完成论文一章 |

抗作弊：1 个 ★★★★★(8) > 8 个 ★(8)。

**加成**：按时完成 +20%、连击 +3%/天上限 50%（v0.2 修订为阶梯式）、高难度(4-5★) +50%，可叠加。

**矿石用途**：购买区域装饰 + 解锁主题皮肤（晨光白免费，其余 7 套用矿石解锁）。矿石只增不减，撤销/删除时不收回。

### 4.2 等级系统

公式：`所需 XP = round(5 × ln(level+1) × level)`，不封顶。节奏：第一年每 2 周升一级，后续每月升一级。允许降级（XP 不足时等级降低）。

**称号**（2 套预设 + 用户自定义）：
- 锻造系：破壳 / 熔炼 / 锤炼 / 锋芒 / 极意 / 铸魂 / 锻石 / 不朽
- 经典 RPG：新手 / 学徒 / 冒险者 / 游侠 / 英雄 / 大师 / 史诗 / 传奇

**淬火重铸（Prestige）**：Lv.51 可重铸，等级归 1 但保留所有数据，记录重铸次数，解锁专属称号。

### 4.3 任务系统

- **状态**：待办 → 进行中 → 已完成（软删除进回收站，30 天清理）
- **难度** 5 级 + 文字描述，**优先级** 4 级（紧急/高/中/低）
- **子任务**：最多 3 层嵌套（PRD 原定 6 层，实现为 3 层）
- **创建**：快速输入（顶部回车）+ 完整抽屉
- **分类选择器**：按钮组，非下拉菜单
- 4-5 星完成需二次确认；完成时卡片向右飞出
- 撤销：Toast (5 秒) + 已完成列表双入口，XP 收回矿石不收回
- 支持隐藏任务（不计入统计）、拖拽排序

### 4.4 习惯系统

- **重复周期** 8 种：每天 / 工作日 / 每周严格 / 每周灵活 / 每两周 / 每月 / 每月固定日期 / 自定义
- **连续计数**：完成 +1、跳过 −10%（最少 −1）、错过减半
- **容错充能**：首次 7 连续得 1 个，之后 14 次得 1 个，最多持有 1 个，仅用于抵消错过
- **灵活次数**：一个周期内可设置目标次数（N/周期），达到后结算 XP
- 状态：active / completed_today / paused（冻结连续）/ mastered / archived

### 4.5 角色状态（5 种温和状态）

| 状态 | 触发条件 | 文案 |
|------|---------|------|
| 活跃 | 近期完成任务 | — |
| 充电中 | 1-3 天未活跃 | 好好充电，随时准备出发 |
| 休憩 | 3-7 天未活跃 | 好好休息，蓄力再出发 |
| 远行 | 7 天以上未活跃 | 无论多久，这里都等你回来 |
| 回归 | 中断后重新活跃 | 欢迎回来，冒险者 |

### 4.6 设计系统

- **8 套主题**：亮色 4 套（晨光白★默认/薄荷纸/羊皮卷/云石灰）+ 暗色 4 套（锻铁紫/矿石青/烈焰铸/翠林绿）
- 字体：中文 Noto Sans SC，英文 Inter，数字 Inter SemiBold + tabular-nums
- 间距 6 级（4/8/12/16/24/32px），圆角 5 级（4/8/12/16/9999px）
- 颜色通过 CSS 变量实现主题切换，组件内不硬编码色值

### 4.7 反馈动画系统

`FeedbackContext` 统一调度：
- XPFloat：浮动数字，1.1s
- LevelUpCelebration：升级弹窗，1.2s 延迟
- OreFloat：矿石浮动
- StreakMilestonePopup：连续里程碑
- PrestigeModal：淬火重铸确认
- BadgeNotification：徽章解锁通知

## 五、技术实现

### 5.1 技术栈

| 技术 | 版本 | 用途 |
|------|------|------|
| Electron | 41 | 桌面应用框架，无边框窗口 + IPC |
| React | 19 | UI 框架，函数式组件 + Hooks |
| TypeScript | 5.9 | 严格模式，零 `any` |
| Zustand | 5 | 状态管理，persist 中间件 |
| Framer Motion | 12 | 动画系统 |
| Tailwind CSS | 3 | 样式，配合 CSS Variables 主题 |
| localStorage | — | 数据持久化（~5MB 上限） |

### 5.2 架构分层（单向依赖）

```
types/       → 纯类型定义
engines/     → 纯函数业务逻辑（5个：xp/level/prosperity/badge/habit）
stores/      → Zustand 状态（9个 + accountManager）
components/  → React 组件
i18n/        → zh.ts / en.ts（类型安全，609 key × 2 语言）
utils/       → 工具函数
```

**铁律**：engines 永远不 import stores。

### 5.3 关键技术细节

- **数据迁移**：`onRehydrateStorage` 处理字段补丁（`?? defaultValue`）和版本兼容
- **多账号**：`getStoragePrefix()` 返回 `'anvilite'`（默认）或 `'anvilite-{uuid}'`，切换账号 `window.location.reload()`，默认账号 prefix 与旧版 key 完全一致，无需迁移
- **仪表盘网格**：24 列等宽 + 44px 行高 + 8px 间距，`snapPos()`/`snapSpan()` 对齐，布局持久化
- **世界地图画布**：自定义无限画布，`transform: translate(x,y) scale(s)` + 鼠标拖拽 + 滚轮缩放
- **雷达图**：纯 SVG，极坐标计算，3+ 维度雷达 / 1-2 维度自动降级为柱状图，自适应多行标签
- **Framer Motion 分层**：CSS 定位用外层 `<div>`，`motion.div` 只做动画属性（避免 transform 冲突）
- **习惯重置**：`onRehydrateStorage` 检查 `lastCompletedAt` 日期，跨日重置 `completed_today → active`

### 5.4 已解决的技术难点

| 问题 | 解决方案 |
|------|----------|
| Framer Motion transform 与 CSS transform 冲突 | 分层包裹（外层 div 做定位，motion.div 做动画） |
| 编辑模式点击白屏 | React state 替代直接 DOM cursor 操作 |
| `overflow: hidden` 裁剪卡片左边框 | 列容器加 `paddingLeft: 4` |
| 英文雷达图标签逐字换行 | `splitLabel()` 检测英文后按空格分词 |
| 习惯 `completed_today` 跨日不重置 | `onRehydrateStorage` 日期比较 |
| 周弹性习惯完成计数永为 0 | 新增 `weeklyCompletionCount` + 周一重置 |
| 删除任务事件查找用硬编码中文 | 统一使用 i18n 函数生成 title |
| deleteAccount 正则反向匹配 | 简化为 UUID 前缀检测 |

## 六、v0.1 已知限制（v0.2 待解决）

| 限制 | 影响 |
|------|------|
| 无云同步 | 数据仅在本机 |
| 无 Error Boundary | 组件异常导致白屏 |
| localStorage ~5MB 上限 | 大量数据可能溢出 |
| 无数据导出/导入 | 无法备份或迁移 |
| 长期运行不重启时习惯不刷新 | 跨午夜后 completed_today 不重置 |
| 周弹性习惯 N>1 无精确周计数 | 完成 1 次后仍每天显示 |
| 淬火重铸系统（PRD 已设计） | v0.1 未完整实现 |
